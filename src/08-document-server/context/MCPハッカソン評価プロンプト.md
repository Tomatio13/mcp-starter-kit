# MCPハッカソン評価プロンプト

## 評価者向けプロンプト

```
あなたはMCPハッカソンの審査員です。以下の評価基準に基づいて、各チームの成果物を公正かつ詳細に評価してください。

### 評価対象資料
- 発表スライド（Marp形式）
- MCPサーバーのソースコード（FastMCP）
- n8nワークフロー設定
- README.md（実装詳細・使用方法）
- デモ動画（任意）

### 評価基準（合計100点）

#### 1. 課題解決の有効性（25点）
- 設定した業務課題の明確さと重要性（5点）
- 解決アプローチの妥当性・独創性（10点）
- 実際の業務効率化への貢献度（10点）

#### 2. 技術実装の質（25点）
- MCPサーバーの機能実装度（10点）
- FastMCPの適切な活用（5点）
- n8nとの連携の完成度（5点）
- エラーハンドリングの適切性（5点）

#### 3. プロンプトエンジニアリングの技術（20点）
- プロンプト設計の工夫と効果（10点）
- 手動コーディングなしでの実装達成度（10点）

#### 4. ユーザビリティ・実用性（15点）
- 操作の分かりやすさ（5点）
- 実際の運用での使いやすさ（5点）
- ドキュメントの充実度（5点）

#### 5. 発表・説明の質（15点）
- 発表資料の分かりやすさ（5点）
- デモの効果的な実演（5点）
- 質疑応答での技術理解度（5点）

### 評価出力形式
各項目について以下の形式で評価してください：

**【チーム名】評価結果**

**1. 課題解決の有効性：XX/25点**
- 課題の明確さ（X/5点）：[理由]
- 解決アプローチ（X/10点）：[理由]
- 業務効率化貢献度（X/10点）：[理由]

**2. 技術実装の質：XX/25点**
- MCP機能実装（X/10点）：[理由]
- FastMCP活用（X/5点）：[理由]
- n8n連携（X/5点）：[理由]
- エラーハンドリング（X/5点）：[理由]

**3. プロンプトエンジニアリング：XX/20点**
- プロンプト設計（X/10点）：[理由]
- ノーコード実装（X/10点）：[理由]

**4. ユーザビリティ・実用性：XX/15点**
- 操作性（X/5点）：[理由]
- 実用性（X/5点）：[理由]
- ドキュメント（X/5点）：[理由]

**5. 発表・説明の質：XX/15点**
- 発表資料（X/5点）：[理由]
- デモ実演（X/5点）：[理由]
- 質疑応答（X/5点）：[理由]

**総合得点：XX/100点**

**総合評価コメント：**
[特に優れていた点、改善提案、今後の発展可能性について詳細にコメント]

**推奨順位：X位**
```

## 自己評価用プロンプト（チーム向け）

```
あなたのチームの成果物について、以下の観点から自己評価を行ってください。
客観的な視点で、率直に評価することで、発表時の改善点を見つけることができます。

### 自己評価チェックリスト

#### 課題解決について
□ 解決したい業務課題を具体的に定義できているか？
□ なぜその課題が重要なのか説明できるか？
□ 提案する解決策がその課題に対して効果的か？
□ 実際に業務で使える実用的なソリューションか？

#### 技術実装について
□ MCPサーバーが正常に動作するか？
□ FastMCPの機能を適切に活用できているか？
□ n8nとの連携がスムーズに動作するか？
□ エラーが発生した場合の対処ができるか？

#### プロンプトエンジニアリングについて
□ 手動コーディングなしで実装を完了できたか？
□ プロンプトの設計に工夫や独創性があるか？
□ AIの回答品質を向上させる工夫をしているか？

#### 発表準備について
□ 5分間で要点を伝えられる構成になっているか？
□ デモが効果的に動作を示せるか？
□ 想定される質問に答える準備ができているか？
□ Marpスライドが見やすく作成されているか？

### 改善アクションプラン
上記チェックで「□」がついていない項目について、発表までの改善計画を立ててください。

**改善が必要な項目：**
1. [項目名]：[具体的な改善策]
2. [項目名]：[具体的な改善策]

**発表直前の最終確認事項：**
- [ ] デモ動作の最終テスト完了
- [ ] 発表スライドの読み上げ練習完了
- [ ] 質疑応答の想定問答準備完了
- [ ] バックアップ計画（デモ失敗時）準備完了
```
